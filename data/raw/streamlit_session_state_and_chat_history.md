# Streamlit Session State and Chat History

## When This Matters

Use this document when the Streamlit UI behaves correctly on the first request but breaks on rerun, loses chat history, clears the wrong state, or shows inconsistent usage and export data. This app is intentionally rerun-driven. The stable behavior comes from storing only the right values in `st.session_state` and rebuilding the visible UI from that state on each rerun.

The main session keys are:

- `conversation_history`
- `request_timestamps`
- `selected_chat_model`
- `kb_rebuild_feedback`

If one of these keys is missing or reset at the wrong time, the chat flow becomes confusing very quickly.

## Recommended Pattern

Initialize session state once at the top of `main()` before reading inputs:

- `conversation_history` starts as `[]`
- `request_timestamps` starts as `[]`
- `selected_chat_model` starts as `settings.default_chat_model`

Then follow this rerun-safe sequence:

1. Render sidebar controls with `render_help_section(...)`.
2. Re-render previous turns with `render_latest_turn()`.
3. Read one new question from `st.chat_input(...)`.
4. Validate and process that question.
5. Convert the backend result into a turn record with `build_turn_record(...)`.
6. Append the turn to `st.session_state["conversation_history"]`.
7. Call `st.rerun()` so the chat re-renders from stored history.

This pattern matters because the app does not keep a separate rendered-chat cache. The UI is reconstructed from `conversation_history` every time.

Each turn record stores only UI-relevant data:

- `query`
- `answer`
- `used_context`
- `sources`
- `tool_result`
- `official_docs_result`
- `usage`

That design keeps the backend stateless per question while the frontend still feels conversational.

## Common Failure Modes

- Chat history disappears after sending a question.
  Usually this means the code generated a response but never appended `build_turn_record(...)` to `conversation_history` before rerunning.

- Clear chat resets too much.
  In the current app, `Clear chat` should only reset `st.session_state["conversation_history"]`. It should not wipe `request_timestamps`, cached resources, or the selected model.

- Export output does not match the visible chat.
  Exports are built from `conversation_history`, not from the current rendered widgets. If turn records are missing `tool_result`, `official_docs_result`, or `usage`, the exports will look incomplete.

- Usage display looks wrong.
  `format_request_usage_label(...)` reports `No LLM usage` for tool results, `Usage unavailable` when a grounded or official-docs answer has no token metadata, and a detailed token line only when real usage metadata exists.

- Model selection seems inconsistent across reruns.
  The selected model is stored in `selected_chat_model` and validated by `validate_selected_chat_model(...)`. If you bypass that path, the UI and the actual `ChatOpenAI` client can drift.

- KB rebuild feedback disappears immediately.
  The sidebar stores one short message in `kb_rebuild_feedback`, shows it on the next run, and then pops it. That is deliberate.

## Implementation Notes

The sidebar controls combine several stateful features without turning the app into a full multi-page UI:

- chat model selection via `selected_chat_model`
- clear chat
- export format selection and download
- knowledge-base rebuild status and feedback
- session-level usage totals through `format_session_usage_label(...)`

Conversation export supports four formats:

- Markdown via `build_conversation_markdown(...)`
- JSON via `build_conversation_json(...)`
- CSV via `build_conversation_csv(...)`
- PDF via `build_conversation_pdf(...)`

All of them are derived from a stable serialized snapshot generated by `build_conversation_snapshot(...)`. That is why export remains consistent even though Streamlit reruns the script repeatedly.

For official-docs responses, the UI treats them as their own path. `get_response_type_label(...)` returns `Official docs answer`, and the app renders a dedicated `Official Docs Result` expander instead of mixing those results into normal knowledge-base sources.
